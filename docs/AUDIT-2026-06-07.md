## P0 — Critical (release blockers)

### P0-1 [CONFIRMED] `created_by` ownership regression: filters/owner-checks still use `user.email`, but the column now stores the Clerk id
The 25.5 migration changed `created_by` to hold the Clerk `sub` (`user.id`, e.g. `user_xxx`) — `secureEntity.create` stamps `[ownerField]: user.id` (`src/lib/secureEntity.js:37`). But the app still queries/compares `created_by` against `user.email` everywhere. Result: a registered user's own content reads as empty / not-owned.

Affected (all `created_by` vs `user.email`):
- `src/pages/Library.jsx:72` — `Book.filter({ created_by: hookUser.email })` → **Library always empty.**
- `src/pages/Home.jsx:161` — `Book.filter({ created_by: user.email }, "-created_date", 10)` → **Home "recent books" always empty.**
- `src/pages/Profile.jsx:121` — `Book.filter({ created_by: user.email }, ...)` → **Profile book list + derived stats always empty.**
- `src/hooks/useGamification.js:186` — `Book.filter({ created_by: currentUser.email })` → **XP / gamification book counts wrong.**
- `src/pages/Characters.jsx:58` — `data.filter(c => c.created_by === user.email)` → **user's characters filtered out (empty).**
- `src/pages/BookView.jsx:650` — `book.created_by === user?.email` → **owner never recognized → owner-only controls hidden from the real owner.**
- `src/pages/BookCreation.jsx:170` — `setIsOwner(bookData.created_by === currentUser.email)` → **isOwner always false.**
- `src/pages/Feedback.jsx:96` — `bookData.created_by === user.email` → **owner check broken.**

Fix (one-liner each): compare/filter against `user.id` (Clerk id), not `user.email`. For owner-only lists the filter can also simply be dropped (RLS already scopes reads to the owner by `sub`); the equality `isOwner` checks must use `user.id`.

### P0-2 [CONFIRMED] Production `service_role` Supabase key hardcoded & committed to git
`scripts/setup-supabase-storage.mjs:13` contains a full `service_role` JWT (role `service_role`, ref `furviizyohryyqubosut`, exp 2089), git-tracked (`git ls-files` confirms). This key **bypasses all RLS** — it nullifies the entire RLS lockdown for anyone with repo read access.
Fix: rotate the service-role key in Supabase immediately; move to `process.env.SUPABASE_SERVICE_ROLE_KEY`; purge from history (or accept rotation as mitigation). This is the single most urgent item.

## P1 — High (broken feature / broken integration for real users)

### P1-1 [CONFIRMED] Creem webhook HMAC verified over a re-stringified body → all valid webhooks rejected
`api/webhooks/creem.js:64-66` computes the signature over `JSON.stringify(req.body)`. Vercel auto-parses the JSON body, so re-stringifying produces different bytes (key order/whitespace) than what Creem signed (raw body). `crypto.timingSafeEqual` → mismatch → 401 for every legitimate webhook. There is **no `export const config = { api: { bodyParser: false } }`** in the file (grep-confirmed). Net effect: subscriptions never get written/updated → paid users stay on `free`.
Fix: disable the body parser for this route and read the raw request stream for HMAC, then `JSON.parse` the raw string for the event.

### P1-2 [CONFIRMED] Leaderboard shows only the current user, with a raw Clerk id as the name
`src/pages/Leaderboard.jsx:74` `Book.list("-created_date", 200)` now returns ONLY the caller's books (RLS scopes base `books` to owner) → the "leaderboard" can never contain other users. Compounded by P0-1: `email = book.created_by` is now a Clerk id (`user_xxx`), so `email.split("@")[0]` (line 96) yields the whole id as the display name, and `email === user.email` (line 102) is always false. `created_by_name` (the intended fallback) is a table column that the app never writes (grep-confirmed) so it's always null.
Fix: source the leaderboard from a data set that legitimately exposes other users (e.g. the sanitized `public_books` view / community, or a dedicated leaderboard RPC/aggregate), and resolve display names via the `profiles` directory keyed on Clerk id — not `created_by.split('@')`.

## P2 — Medium (suspect / edge-case / cosmetic)

### P2-1 [SUSPECT] `ensureProfile` upsert with `ignoreDuplicates:true` never updates display_name/avatar on later logins
`src/lib/profiles.js:30-38`. `onConflict:'clerk_id', ignoreDuplicates:true` means after the first insert, subsequent sign-ins are a no-op even when the user changed their Clerk display name / avatar. The `profiles` directory drifts stale (affects follow/notify display + any leaderboard fix that reads profiles). Intentional re: the immutable-email trigger, but it also blocks legit display_name/avatar refresh.
Fix: upsert non-email fields without `ignoreDuplicates` (omit `email` from the update payload so the immutable-email trigger is not tripped), or use a targeted `update` of display_name/avatar.

### P2-2 [SUSPECT] Image-gen data-URI fallback persisted into `Page.image_url`
`src/integrations/Core.js:71` returns `data:<mime>;base64,...` when the Supabase Storage upload fails; `BookWizard` then saves that into `Page.create({ image_url })` (`src/pages/BookWizard.jsx:905,916`). A full illustration data URI is ~1–3 MB; persisting it per page bloats rows and can hit PostgREST/row limits. Low likelihood in practice — the storage bucket insert policy allows `anon,authenticated` and the Supabase client injects the Clerk `supabase`-template JWT, so uploads should normally succeed — but the failure mode writes large blobs into the DB silently. Worth a guard (skip persisting if the URL is a `data:` URI, or mark the page failed instead).

### P2-3 [LOW] Dead/duplicated public-route checks (case-mismatch, currently harmless)
`src/App.jsx:69-80` `isPublicRoute` includes lowercase `'/pricing'` and `'/contact'` while `createPageUrl` emits capitalized paths (`/Pricing`, `/Contact`, `src/utils/index.ts:9`). These lowercase checks never match; public access is saved only by `PUBLIC_PAGES.has(currentPath)` (capitalized Set). Harmless today but fragile — any future reliance on the lowercase branch will silently fail. Recommend deleting the redundant lowercase string checks and relying on the Set.

### P2-4 [LOW] `min-h-screen` used in ~12 JSX files (project rule prefers `h-dvh`/`min-h-dvh`)
e.g. `src/pages/SignIn.jsx:18`. On mobile browsers the dynamic toolbar makes `100vh` overshoot the viewport. Cosmetic; batch-swap to `min-h-dvh` per the repo's own CLAUDE.md / global UI rules.
